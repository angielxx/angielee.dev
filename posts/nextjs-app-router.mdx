## App Router란

Next.js 13부터 도입된 새로운 라우팅 시스템이다. `app/` 디렉토리를 기반으로 하며, React Server Components를 기본으로 사용한다.

기존 `pages/` 기반 라우팅과 함께 사용할 수 있어 점진적인 마이그레이션이 가능하다.

## Server Component vs Client Component

App Router의 핵심은 **Server Component**다.

### Server Component

- 기본값: `app/` 디렉토리의 모든 컴포넌트
- 서버에서 렌더링되어 HTML로 전달
- `useState`, `useEffect` 등 훅 사용 불가
- 데이터베이스, 파일 시스템에 직접 접근 가능
- 번들 크기에 포함되지 않음

```tsx
// app/posts/page.tsx — Server Component (기본값)
import { getAllPosts } from "@/lib/posts";

export default async function PostsPage() {
  const posts = await getAllPosts(); // 서버에서 직접 실행
  return <ul>{posts.map(p => <li key={p.slug}>{p.title}</li>)}</ul>;
}
```

### Client Component

`"use client"` 지시어를 파일 최상단에 선언하면 된다.

```tsx
"use client";

import { useState } from "react";

export default function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
```

## 레이아웃 시스템

`layout.tsx`는 자식 페이지들이 공유하는 UI다. 리렌더링 없이 유지된다.

### 중첩 레이아웃

```
app/
├── layout.tsx        ← 전체 공통 레이아웃
├── blog/
│   ├── layout.tsx    ← 블로그 전용 레이아웃
│   └── page.tsx
```

## 데이터 페칭

App Router에서는 async/await를 컴포넌트에서 바로 사용한다.

```tsx
export default async function Page() {
  const data = await fetch("https://api.example.com/data");
  const json = await data.json();
  return <div>{json.title}</div>;
}
```

### 캐싱 전략

- `cache: "force-cache"` — 기본값, 빌드 시 캐시
- `cache: "no-store"` — 매 요청마다 새로 fetch
- `next: { revalidate: 60 }` — ISR, 60초마다 재검증

## 마무리

App Router는 처음에 낯설지만, Server Component 개념만 잡으면 나머지는 자연스럽게 따라온다. 다음 편에서는 MDX 기반 블로그를 직접 구현해보겠다.
